# 版本控制

集中式版本控制工具：SVN，CVS，VSS……

有一个单独的集中管理的服务器，保存所有文件的修订版本。每个人可以一定程度地看到其他人正在做什么，管理员也可以管理每个开发者的权限。如果中央服务器存在单点故障，则所有人都将无法协同工作。

分布式版本控制工具：Git……

客户端把代码仓库完整地镜像下来，每个客户端的每一次文件提取操作都是一次对整个文件仓库的完整备份。

# 工作机制

工作区 -> 暂存区 -> 本地库 -> 远程库

代码托管中心是基于网络服务器的远程代码仓库，即远程库。

# 常用命令

命令名称 命令
设置用户签名 git config --global user.name <>
git config --global user.mail <>
初始化本地库 git init
查看本地库状态 git status
添加到暂存区 git add <>
提交本地库 git commit -m “” <>
查看版本信息 git reflog
查看详细版本信息 git log
版本穿梭 git reset --hard

.git 文件夹下存放了本地分支对象、仓库信息、日志文件等。

# 分支

在版本控制中，同时推进多个任务，为每个任务创建新的分支。分支底层是由指针实现的。

同时并行推进多个功能开发，提高开发效率。各分支在开发过程中一个分支的失败对其他分支没有任何影响。

分支常用命令
创建分支 git branch <分支名>
查看分支 git branch -v
切换分支 git checkout <>
合并到所在分支上 git merge <>

冲突合并：合并分支时两个分支对同一个位置有两套不同的修改。合并时要手动修改、添加到暂存区、执行提交（不再带文件名）。

各分支都是指向具体版本的指针，当前所在的分支是由 Head 决定的。创建分支的本质是创建一个新的指针。

# 团队协作

团队协作使用代码托管中心（远程库）。

使用 push、clone、pull 进行团队内协作。

使用 fork 进行团队间协作，fork 将其他远程库中的项目拷贝到此远程库；使用 Pull request 将 fork 来的项目对原项目请求合并。

# GitHub 操作

常用操作
创建远程库别名 git remote add <别名> “远程库地址”
推送 git push <远程库别名或地址> < 分支 >
拉取 git pull <别名或地址> < 分支 >
克隆 git clone <地址>

使用库的 Manage Access 进行权限管理。

跨团队协作：使用 Fork 将其他项目仓库 “叉” 到此仓库。使用 Pull request 将修改后的代码提交到原仓库。
